# 15 — Shipping It: Multi-Title Architecture & Build System

## What this covers / why it matters

You have a VR mod that works on one game. Now you want it to work on four games in the same engine family — and ideally to share its hard-won core (injection, D3D hooks, the imgui overlay, the VR runtime, the frame timeline) with an entirely *different* engine that someone else is porting. This guide is about the boring-but-load-bearing scaffolding that makes that possible: how the project is laid out so one codebase produces many injectable DLLs, how per-title data (offsets, settings, reclass structs) is externalized so a new game is a folder and not a fork, and how the build (CMake + presets + git submodules) wires it all together and drops a correctly-named `dxgi.dll` next to each game's `.exe`.

The three reference projects make three different bets here, and the differences are instructive. REFramework ships **one universal DLL** that detects the game at runtime. `anvilengine2vr` ships **one DLL per title**, generated by a CMake loop over a list of game names. `starfield2vr` is a **single-game** project that still inherits the shared-core split. All three sit on top of the same private `vrframework` core. Understanding *why* each made its choice is the point — because on your new engine you'll make the same choice.

---

## 1. The three-layer split, made concrete in folders

Recall the mental model the whole guide series uses:

- **Layer 1 — Universal core** (`vrframework`): injection, D3D hooks, imgui overlay, the `Mod` system, the VR runtime, the frame timeline. Reusable across *engines*.
- **Layer 2 — Engine adapter**: per-engine hooks — frame pacing, projection/view injection, HUD, input. Reusable across *titles in one engine*.
- **Layer 3 — Per-game data**: offsets/patterns, per-title settings, reclass structs. Unique to each *title*.

The filesystem mirrors this exactly. The decisive design move is that **Layer 1 lives in its own git repository** and is pulled into each port as a submodule. That is what lets `starfield2vr` and `anvilengine2vr` — two unrelated engines maintained as separate forks — share a single evolving core without copy-paste drift.

```
anvilengine2vr/
├─ extern/
│  └─ vrframework/          ← Layer 1, git submodule (the shared core)
├─ src/                     ← Layer 2, the Anvil engine adapter (shared by all 4 titles)
├─ games/                   ← Layer 3, one folder per title
│  ├─ Odyssey/
│  ├─ Origins/
│  ├─ valhalla/
│  └─ Mirage/
├─ imgui/imconfig.hpp       ← project-local imgui config injected into the core
├─ CMakeLists.txt
├─ CMakePresets.json
└─ .gitmodules
```

Each game folder is self-similar (`anvilengine2vr/games/valhalla/`):

```
games/valhalla/
├─ ModConfig.h / ModConfig.cpp   ← per-title settings struct + Mods registration
├─ engine/
│  ├─ EngineCameraModule.h       ← title-specific camera/view injection
│  ├─ EngineRendererModule.h     ← title-specific render hooks
│  ├─ EngineEntry.h              ← the per-title Mod entry point
│  ├─ EngineTwicks.h
│  └─ memory/offsets.h           ← Layer 3 proper: patterns + fallback offsets
└─ sdk/
   ├─ ACValhalla.exe.h           ← reclass-style game structs for this exe
   └─ ACVRGfxContext.h
```

Starfield, being a single game, collapses Layers 2 and 3 into one tree (`starfield2vr/src/CreationEngine/`, with `src/CreationEngine/memory/offsets.h` holding the Layer 3 data). It still consumes the same `extern/vrframework` submodule. The lesson: **the split between Layer 1 and Layers 2/3 is non-negotiable and is enforced by a repository boundary; the split between Layers 2 and 3 is a convenience you adopt only once you have more than one title.**

## 2. Externalizing the core as a submodule

Both ports declare the core identically. From `anvilengine2vr/.gitmodules:1`:

```ini
[submodule "extern/vrframework"]
	path = extern/vrframework
	url = https://github.com/mutars/vrframework.git
```

Starfield adds a second submodule for its address library (`starfield2vr/.gitmodules:1`):

```ini
[submodule "extern/CommonLibSF"]
	path = extern/CommonLibSF
	url = https://github.com/mutars/CommonLibSF.git
[submodule "extern/vrframework"]
	path = extern/vrframework
	url = https://github.com/mutars/vrframework.git
```

That second entry is itself a multi-title lesson. `CommonLibSF` is Starfield's reflection/address-library dependency (the Creation Engine analogue to RE Engine's type system). It's pinned as a submodule rather than vendored so the port tracks upstream fixes. Anvil needs no such library — it resolves everything by **pattern scan** (`offsets.h`), which is why its `.gitmodules` has only the one entry. The dependency surface *is* part of the engine adapter.

### Why a submodule and not a copy

The submodule pins an exact commit. When the core gains a feature — say, a better `FrameTimeline` or a new `IEngineAdapter` SPI method — each port bumps the submodule pointer on its own schedule and rebuilds. No file ever gets copied between repos, so the two engine adapters can't silently diverge from the core's contract. This is the whole reason the `vrframework` repo exists as a separate thing (see its `README.md` and `PORTING.md` for the intended `IEngineAdapter` / `FrameTimeline` / `StereoView` SPI that adapters implement against).

### Cloning

Because the core is a submodule, a fresh clone is a two-step operation:

```powershell
git clone https://github.com/mutars/anvilengine2vr.git
cd anvilengine2vr
git submodule update --init --recursive
```

Forgetting `--recursive` is the single most common first-build failure: CMake's `FetchContent_Declare(... SOURCE_DIR .../extern/vrframework)` will point at an empty directory and the configure step dies with a missing `CMakeLists.txt`.

## 3. How the core is consumed by CMake

The core is brought in not over the network but from the checked-out submodule, via `FetchContent` pointed at a local `SOURCE_DIR`. Identical in both ports — `anvilengine2vr/CMakeLists.txt:20`:

```cmake
set(IMGUI_USER_CONFIG_PATH "${CMAKE_CURRENT_SOURCE_DIR}/imgui/imconfig.hpp")
FetchContent_Declare(vrframework SOURCE_DIR ${CMAKE_CURRENT_SOURCE_DIR}/extern/vrframework)
FetchContent_MakeAvailable(vrframework)
```

Two things are happening:

1. **`IMGUI_USER_CONFIG_PATH`** is set *before* the core is made available. The core's imgui build reads this variable to inject a project-local `imconfig.hpp` (each port keeps one at `imgui/imconfig.hpp`). This is the seam that lets a shared core render a per-port-customized overlay without the core knowing anything about the port. It's the same trick REFramework uses inline via `IMGUI_USER_CONFIG` (`REFramework/cmake.toml:124`).
2. **`FetchContent_MakeAvailable`** runs the core's own `CMakeLists.txt`, which defines the `vr_framework` target (and its `vr_framework::vr_framework` alias).

The port then defines an `INTERFACE` library called `common` and threads it *into* the core — `anvilengine2vr/CMakeLists.txt:24`:

```cmake
add_library(common INTERFACE)
target_link_libraries(vr_framework PRIVATE common)
```

`common` is the configuration bus. Compile definitions added to `common` (`USE_DXGI_HOOK`, `HOOK_XINPUT`, `SIGNATURE_SCAN`, …) propagate into the core build, so the same core source compiles differently per port. Starfield uses this aggressively (`starfield2vr/CMakeLists.txt:58`):

```cmake
target_compile_definitions(common INTERFACE DISABLE_WINDOW_SIZE_FIXES)
target_compile_definitions(common INTERFACE HOOK_XINPUT)
target_compile_definitions(common INTERFACE HOOK_DX12_FIRST)
target_compile_definitions(common INTERFACE GLM_ENABLE_EXPERIMENTAL)
```

`HOOK_DX12_FIRST` matters because Starfield is D3D12-only; telling the core to install the D3D12 hook first avoids a wasted D3D11 detection pass. The port configures the shared core through `common`; the core never hard-codes anything game-specific.

## 4. The per-title build loop (Anvil's multi-DLL pattern)

This is the heart of multi-title shipping. Anvil declares its titles as a plain list (`anvilengine2vr/CMakeLists.txt:9`):

```cmake
set(GAME_TARGETS Odyssey Origins Valhalla Mirage)
```

…and at the bottom loops over them (`anvilengine2vr/CMakeLists.txt:163`):

```cmake
foreach(GAME_TARGET IN LISTS GAME_TARGETS)
  configure_game_target(${GAME_TARGET})
endforeach()
```

`configure_game_target()` is a function that, for each title, builds one `SHARED` library out of **the shared `src/` adapter plus that title's `games/<Title>/` folder** (`anvilengine2vr/CMakeLists.txt:47`):

```cmake
function(configure_game_target GAME_TARGET)
  set(TARGET_NAME "anvil_vr_${GAME_TARGET}")
  set(GAME_DIR "${CMAKE_CURRENT_SOURCE_DIR}/games/${GAME_TARGET}")

  file(GLOB_RECURSE headers
      ${CMAKE_CURRENT_SOURCE_DIR}/src/*.h  ${CMAKE_CURRENT_SOURCE_DIR}/src/*.hpp
      ${GAME_DIR}/*.h ${GAME_DIR}/*.hpp)
  file(GLOB_RECURSE sources
      ${CMAKE_CURRENT_SOURCE_DIR}/src/*.cpp ${GAME_DIR}/*.cpp)

  add_library(${TARGET_NAME} SHARED ${headers} ${sources})
  target_link_libraries(${TARGET_NAME} PUBLIC common)
  target_link_libraries(${TARGET_NAME} PUBLIC vr_framework::vr_framework)

  target_include_directories(${TARGET_NAME} PRIVATE
      $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/src>
      $<BUILD_INTERFACE:${GAME_DIR}>)
  ...
```

The mechanism is worth dwelling on:

- **`src/` is shared, `games/<Title>/` is title-specific**, and the `GAME_DIR` is on the include path. Code in `src/` does `#include <engine/EngineEntry.h>` and resolves to *that title's* `EngineEntry.h` because the per-game folder is on the include path. The same source line compiles against four different sets of offsets and structs across four targets. This is compile-time polymorphism by include path — crude, but it produces four lean DLLs with zero runtime game-detection branching.
- Each target gets its own `vr_framework::vr_framework` link, but the core is built **once** and reused; only the thin per-title library is recompiled per game.
- The `OUTPUT_NAME` is set globally from `HOOK_METHOD` (see §6) so every title's DLL is named `dxgi.dll`, distinguished only by output directory.

Contrast this with **REFramework's universal-DLL bet**. Its `cmake.toml` defines a single `REFramework` target and a single `REFrameworkSDK` (`REFramework/cmake.toml:198` and `:216`) with the comment `# ── Single unified DLL (replaces all per-game targets) ──`. REFramework detects the RE Engine game at runtime via the engine's reflection system, so one binary serves Resident Evil, Monster Hunter, and the rest. The trade:

| | REFramework | anvilengine2vr | starfield2vr |
|---|---|---|---|
| Shipping unit | one universal DLL | one DLL **per title** | one DLL (single game) |
| Game selection | runtime reflection | compile-time (include path) | n/a |
| Per-game data | reflection at runtime | `games/<Title>/offsets.h` + `sdk/` | `src/CreationEngine/.../offsets.h` |
| New title cost | often *zero* (reflection covers it) | add a folder + one line in `GAME_TARGETS` | new project/fork |
| Best when | engine exposes rich reflection | engine needs hand-found offsets per exe | only one title exists |

The rule of thumb: **if the engine hands you reflection, go universal like REFramework; if you must pattern-scan per exe, go per-title like Anvil.** Anvil can't be universal because each Anvil exe (Odyssey vs. Valhalla vs. Mirage) has different byte patterns and struct layouts — there's no runtime type system to ask.

## 5. Layer 3: externalizing offsets and per-title settings

Two kinds of data vary per title, and both are pushed out to `games/<Title>/`.

### Offsets / patterns

`games/<Title>/engine/memory/offsets.h` is the entire per-title "address book." Each entry pairs a **byte pattern** (resilient to patches) with a **fallback absolute offset** (fast when the exe is unchanged). From `anvilengine2vr/games/valhalla/engine/memory/offsets.h:13`:

```cpp
inline uintptr_t begin_render_frame_fn_addr()
{
    static const auto pattern = "40 53 48 83 EC 20 48 8B D9 48 8B 0D ? ? ? ? E8 ? ? ? ? 48 8B 0D";
    static auto addr = FuncRelocation("begin_render_frame_fn_addr", pattern, 0x10f4db0);
    return addr;
}
```

Every hook the engine adapter installs (frame pacing, projection calc, view update, the TAA disable) is looked up through one of these functions. Because the *names* are identical across titles (`begin_render_frame_fn_addr`, `calc_projection_fn_addr`, `update_views_fn_addr`) but the *patterns and offsets differ*, the shared `src/` adapter code is title-agnostic — it calls `memory::begin_render_frame_fn_addr()` and the right address resolves at compile time via the include path. **Offsets are data, not code, and they live with the title.**

Starfield, having `CommonLibSF` + Reflex markers, leans less on raw patterns but keeps the same principle: title data in `src/CreationEngine/memory/offsets.h`, separate from the adapter logic.

### Per-title settings struct

User-facing tuning that legitimately differs by game gets its own struct. From `anvilengine2vr/games/valhalla/ModConfig.h:1`:

```cpp
namespace ModSettings
{
    struct ACValhallaSettings
    {
        float hudScaleX{0.4};
        float hudScaleY{0.4 * 1024.0f/1920.0f};
        bool  decoupledPitch{false};
    };
    extern ACValhallaSettings g_ac_valhalla_settings;
}
```

The `hudScaleY` aspect-ratio math and the `decoupledPitch` default are Valhalla-specific calls; Odyssey's struct can differ. Each title also wires up *which mods exist* in its own `ModConfig.cpp` (`anvilengine2vr/games/valhalla/ModConfig.cpp:13`):

```cpp
Mods::Mods() {
    m_mods.emplace_back(VRConfig::get());
    m_mods.emplace_back(VR::get());
    m_mods.emplace_back(EngineEntry::Get());
}
```

`Mods()` is defined per-title even though `VR` and `VRConfig` are shared — because `EngineEntry` is the per-title engine adapter entry point. This is the seam where Layer 3 (this file) plugs the Layer 2 adapter into the Layer 1 `Mod` system. On a new engine, this constructor is the one file you'll always have to write per title.

## 6. The DXGI proxy: how the DLL gets built and named

Injection is a proxy `dxgi.dll` dropped next to the game `.exe`. The build's job is to make the output DLL *named* `dxgi` and *delay-load* the real system DLLs it shadows. The naming is driven by a `HOOK_METHOD` switch, identical in both ports (`anvilengine2vr/CMakeLists.txt:27`):

```cmake
if(NOT DEFINED HOOK_METHOD)
  set(HOOK_METHOD "dxgi")
endif()

if(HOOK_METHOD STREQUAL "dinput8")
  set(OUTPUT_NAME "dinput8")
elseif(HOOK_METHOD STREQUAL "dxgi")
  set(OUTPUT_NAME "dxgi")
  target_compile_definitions(common INTERFACE USE_DXGI_HOOK)
elseif(HOOK_METHOD STREQUAL "PHYSX")
  set(OUTPUT_NAME "PhysX3Cooking_x64")
  target_compile_definitions(common INTERFACE USE_PHYSX_HOOK)
elseif(HOOK_METHOD STREQUAL "inject")
  set(OUTPUT_NAME "vrmod")
endif()
```

The proxy-DLL name is just a *filename* the game's loader will pick up from its own directory before the system copy. `dxgi` is the default because every D3D11/D3D12 game loads `dxgi.dll`; `dinput8` is REFramework's choice (`REFramework/cmake.toml:259` sets `OUTPUT_NAME = "dinput8"`); `PhysX3Cooking_x64` is an escape hatch for games that resolve `dxgi` from system32 first. The `USE_DXGI_HOOK` define flips the *runtime* code path to match the *filename*, so the proxy and the hook agree.

Two more build settings make the proxy viable:

**Delay-load the shadowed imports** (`anvilengine2vr/CMakeLists.txt:104`):

```cmake
LINK_FLAGS "/DELAYLOAD:openvr_api.dll /DELAYLOAD:d3d12.dll /DELAYLOAD:D3DCOMPILER_47.dll"
```

Delay-loading means the DLL doesn't resolve `openvr_api.dll`/`d3d12.dll` at load time — so a machine without an OpenVR runtime, or a game that hasn't created its device yet, still loads the proxy cleanly. The mod resolves these lazily when it actually needs them. REFramework delay-loads the same set plus `openxr_loader.dll` and `d3d11.dll` (`REFramework/cmake.toml:260`) because it supports both APIs and both runtimes in one binary.

**Static runtime** (REFramework, `cmake.toml:22`): in Release it rewrites `/MD` → `/MT` so the injected DLL carries no dependency on the VC++ redistributable being installed on the player's machine. A proxy DLL that needs a redist you can't guarantee is a support nightmare; static-linking the CRT removes the variable.

## 7. Artifact output: where the DLLs land

A mod DLL is useless in `build/`; it has to end up next to the game exe with the right name. Both ports use `POST_BUILD` custom commands to (a) stage a distributable artifact folder and (b) deploy directly to the install for fast iteration.

Anvil does this per title, with a per-game artifact name and Steam path resolved from the target name (`anvilengine2vr/CMakeLists.txt:127`):

```cmake
if(GAME_TARGET STREQUAL "Valhalla")
  set(GAME_ARTIFACT_DIR "assasins_creed_valhalla_vr")
  set(GAME_STEAM_PATH "C:\\UbisoftGames\\Assassin's Creed Valhalla")
elseif(GAME_TARGET STREQUAL "Mirage")
  ...
```

…then stages and deploys (`anvilengine2vr/CMakeLists.txt:141`):

```cmake
add_custom_command(TARGET ${TARGET_NAME} POST_BUILD
  COMMAND ${CMAKE_COMMAND} -E make_directory
    ${CMAKE_CURRENT_SOURCE_DIR}/artifact/Mod${BUILD_NAME}/${GAME_ARTIFACT_DIR}
  COMMAND ${CMAKE_COMMAND} -E copy $<TARGET_FILE:${TARGET_NAME}>
    ${CMAKE_CURRENT_SOURCE_DIR}/artifact/Mod${BUILD_NAME}/${GAME_ARTIFACT_DIR})

add_custom_command(TARGET ${TARGET_NAME} POST_BUILD
  COMMAND ${CMAKE_COMMAND} -E copy $<TARGET_FILE:${TARGET_NAME}>
    "${GAME_STEAM_PATH}\\$<TARGET_FILE_NAME:${TARGET_NAME}>")
```

The output for each title is routed to its own bin directory first (`anvilengine2vr/CMakeLists.txt:108`, `RUNTIME_OUTPUT_DIRECTORY .../bin/${GAME_TARGET}`) so the four `dxgi.dll` files never collide despite sharing a name. The artifact tree comes out as `artifact/ModRelease/assasins_creed_valhalla_vr/dxgi.dll` — a ready-to-zip per-title drop.

Starfield does the same but copies the **PDB** alongside the DLL (`starfield2vr/CMakeLists.txt:145`) and branches its deploy on store: Steam vs. the Xbox/Game Pass install path (`starfield2vr/CMakeLists.txt:156`):

```cmake
if(XBOX_STORE)
  add_custom_command(... COMMAND ${CMAKE_COMMAND} -E copy
    $<TARGET_FILE:${PROJECT_NAME}> "C:\\XboxGames\\Starfield\\Content\\dxgi.dll")
else()
  add_custom_command(... COMMAND ${CMAKE_COMMAND} -E copy
    $<TARGET_FILE:${PROJECT_NAME}> "C:\\...\\Starfield\\$<TARGET_FILE_NAME:${PROJECT_NAME}>")
endif()
```

That `XBOX_STORE` branch is *another* axis of "multi-title": the same exe in two storefronts has two install layouts and (sometimes) needs a different output name. The build models it as a preset toggle rather than a code change.

## 8. CMake presets: making the matrix usable

With per-title targets, two compilers, store variants, and debug/release, the configuration matrix is large. `CMakePresets.json` tames it. Both ports define a hidden `base` preset with the shared cache variables (`anvilengine2vr/CMakePresets.json:9`):

```json
"cacheVariables": {
  "HOOK_METHOD": "dxgi",
  "SIGNATURE_SCAN": "ON",
  "POLYHOOK2_FUNCTION_HOOK": "OFF"
},
"generator": "Ninja",
"binaryDir": "${sourceDir}/build/${presetName}"
```

…hidden `msvc` / `clang-cl` toolchain presets, and then concrete presets that `inherit` a `base` + a toolchain and pin a build type. Starfield's `base` additionally pins `USE_STARFIELD_SDK_LITE: ON` and `USE_DXGI_HOOK: ON` (`starfield2vr/CMakePresets.json:18`), and adds dedicated `*-xbox` presets that set `XBOX_STORE: ON` (`starfield2vr/CMakePresets.json:105`). The visible `buildPresets` are numbered and human-labeled — "1. Release (MSVC)", "6. Release without debug symbols (MSVC) xbox" — so a contributor doesn't have to remember flag combinations:

```powershell
cmake --preset build-release-msvc
cmake --build --preset release-msvc
```

The payoff: the *how-to-build* lives in version control next to the code, the same on every machine, and the multi-title/multi-store complexity is one dropdown deep instead of a wiki page of `-D` flags.

### The SDK-Lite toggle (a vendoring decision in the build)

Starfield's `USE_STARFIELD_SDK_LITE` is worth calling out as a multi-title pattern of its own (`starfield2vr/CMakeLists.txt:27`):

```cmake
if(NOT USE_STARFIELD_SDK_LITE)
  add_subdirectory(extern/CommonLibSF)
  include("extern/CommonLibSF/CommonLibSF/cmake/CommonLibSF.cmake")
else()
  include_directories(${CMAKE_CURRENT_SOURCE_DIR}/sdk-lite/include)
  add_compile_definitions(USE_STARFIELD_SDK_LITE)
endif()
```

The full `CommonLibSF` submodule is heavy and slow to build; `sdk-lite` is a trimmed set of reclass headers checked into the repo for fast iteration. The build lets you choose at configure time. On a new engine you'll face the same fork: pull the whole address-library submodule, or vendor a thin header subset — and you'll want it behind a flag, not hard-coded.

## 9. Applying this to a brand-new engine

Putting it together, here's the checklist for standing up a new port that shares the core:

1. **New repo, add the core as a submodule.** `git submodule add <vrframework-url> extern/vrframework`. Pin a known-good commit. Implement the `IEngineAdapter` / `FrameTimeline` / `StereoView` SPI from `vrframework/PORTING.md` in your Layer 2 `src/`.
2. **Wire CMake to the submodule.** Copy the `FetchContent_Declare(... SOURCE_DIR .../extern/vrframework)` + `FetchContent_MakeAvailable` block and the `common` INTERFACE library. Push your engine's quirks (`HOOK_DX12_FIRST`, `HOOK_XINPUT`, etc.) into `common`, not the core.
3. **Decide universal vs. per-title.** Reflection-rich engine → one DLL, runtime detection (REFramework). Pattern-scanned exes → a `GAME_TARGETS` list and a `configure_game_target()` loop (Anvil). Only one title for now → a single target, but *keep the submodule split* so you can graduate later (Starfield).
4. **Externalize Layer 3.** One `offsets.h` (pattern + fallback offset per function), one per-title settings struct, one per-title `Mods()` constructor, reclass headers under `sdk/`. Put them where the include path makes them resolve per target.
5. **Name and link the proxy.** `OUTPUT_NAME = "dxgi"` (or `dinput8`), `USE_DXGI_HOOK` define to match, `/DELAYLOAD` the runtime DLLs, static CRT (`/MT`) for Release.
6. **POST_BUILD deploy + artifact staging.** Route each target to its own `bin/<Title>` directory (so identically-named DLLs don't collide), copy to a per-title `artifact/` folder, and copy into the live game dir for iteration. Branch on storefront if needed.
7. **Presets for the matrix.** A hidden `base`, hidden toolchain presets, numbered visible build presets. Add store/SDK toggles as cache variables, never as edits.

The deepest point: **the only thing that makes a "new engine" cheap is that Layer 1 is a separately-versioned dependency.** Everything else — the per-title loop, the offsets file, the proxy naming — is mechanical once the repository boundary is in place. Get the submodule split right first; the rest is plumbing.

---

## Key takeaways

- **The Layer 1 / Layer 2-3 boundary is a *repository* boundary.** `vrframework` is a git submodule (`extern/vrframework`) shared by every port; that single fact is what makes reuse across unrelated engines real instead of aspirational.
- **The `common` INTERFACE library is the configuration bus.** Ports inject compile definitions into the shared core through `common` (`HOOK_DX12_FIRST`, `USE_DXGI_HOOK`, `HOOK_XINPUT`) without forking core source — and feed it a project-local imgui `imconfig.hpp` via `IMGUI_USER_CONFIG_PATH`.
- **Universal vs. per-title is dictated by the engine.** REFramework ships one reflection-driven DLL; Anvil loops a `GAME_TARGETS` list through `configure_game_target()` to emit one `dxgi.dll` per exe because each Anvil exe needs its own patterns; Starfield is single-game but keeps the split.
- **Layer 3 is data, and it lives with the title:** `games/<Title>/engine/memory/offsets.h` (pattern + fallback offset), a per-title settings struct (`ACValhallaSettings`), and a per-title `Mods()` constructor that plugs the engine adapter into the core's mod system.
- **The proxy DLL is a build concern:** `OUTPUT_NAME` picks the shadowed filename (`dxgi`/`dinput8`), a matching define flips the runtime path, `/DELAYLOAD` keeps load-time clean, and POST_BUILD commands route per-title outputs to separate dirs and deploy them.
- **Presets encode the matrix** (compiler × build type × store) so the multi-title complexity is one dropdown deep, checked into the repo, identical on every machine.
